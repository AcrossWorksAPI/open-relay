# Resume Project Packet Design

Date: 2026-06-29

## Purpose

`resume-project` is the implementer-facing packet that turns a validated
`review-response` into a concrete continuation brief for Codex or another local
implementer.

It is the missing hop after:

1. Codex creates a `review-request`.
2. The packet is transported to a PR.
3. A reviewer creates a `review-response`.
4. The response is fetched or received locally.

`resume-project` answers: "Given this structured review response, what should
the implementer consider next, and what gates still apply?"

It does not apply fixes, run commands, post to GitHub, merge, publish, or invoke
an agent.

## User Story

As the local implementer, I can run:

```bash
open-relay generate resume-project \
  --response review-response.json \
  --format markdown \
  --output resume.md

open-relay render resume-project.json --template codex \
  --output codex-resume.md
```

The first command creates a validated `resume-project/0.1` packet from the
review response. The second command uses the existing prompt renderer to wrap
that packet for Codex as untrusted context.

## Design Calls

### Build From `review-response`, Not From PR Prose

The producer accepts only a validated `review-response` packet. It does not
parse native GitHub reviews, Markdown comments, or free-form PR discussion.

Rationale: Open Relay is useful because each hop is explicit and
machine-validatable. Native review import can be a later feature, but this slice
should keep the loop packet-native.

### Generate A New Packet, Not Another Prompt Alias

`render --template codex` already creates a prompt from any validated packet.
`resume-project` adds value by creating a machine-validatable continuation
record before that prompt step.

The command is therefore:

```text
open-relay generate resume-project --response <review-response.json> [--format json|markdown] [--output <path>]
```

No `handoff resume-project` or `respond` alias is added in this slice.

### Preserve Reviewer Meaning Without Authoring New Findings

The producer maps review findings into continuation tasks. It does not decide
whether a finding is correct, rewrite the reviewer recommendation, or generate
new implementation steps from prose.

Each task keeps the source finding id, severity, blocking flag, title,
description, evidence, recommendation, and optional location.

### Encode Gates As Protocol-Owned Safety Flags

The packet includes static safety gates:

- preserve unrelated changes;
- require explicit approval for merge;
- require explicit approval for publish;
- require explicit approval for destructive commands.

These flags are generated by Open Relay, not copied from reviewer prose.

They are guardrails for the receiving agent and for humans inspecting the
packet. They are not an enforcement mechanism.

### Keep Prior Verification Separate

The reviewer's verification is copied as `prior_verification`. It is evidence
about what was considered during review. It is not proof that fixes have been
implemented or re-tested.

The producer does not invent a "not run" verification entry and does not run
tests automatically.

## Packet Fields

| Field | Type | Required | Purpose |
| --- | --- | --- | --- |
| `packet_version` | string | Yes | Must be `0.1`. |
| `packet_type` | string | Yes | Must be `resume-project`. |
| `created_at` | string | Yes | ISO-8601 timestamp when the resume packet was created. |
| `resume_from` | object | Yes | Stable reference to the source `review-response`. |
| `target` | object | Yes | Repository, branch, commits, and diff being resumed. |
| `resume_status` | string | Yes | Derived continuation status. |
| `confidence` | string | Yes | Copied from the review response. |
| `summary` | string | Yes | Copied review summary. |
| `tasks` | array | Yes | Continuation tasks derived from review findings; empty when none exist. |
| `reviewed_scope` | object | Yes | Reviewed files and limitations from the response. |
| `prior_verification` | array | Yes | Verification evidence from the response; empty when none exists. |
| `safety_gates` | object | Yes | Protocol-owned continuation gates. |
| `provenance` | array | No | External evidence copied from the response when present. |
| `redactions` | array | Yes | Redactions copied from the response; empty when none exist. |
| `sensitive_data` | object | No | Optional sensitive-data note copied from the response. |
| `next_action` | string | Yes | Copied review next action. |

## Object Shapes

### `resume_from`

| Key | Type | Required | Notes |
| --- | --- | --- | --- |
| `packet_type` | string | Yes | Must be `review-response`. |
| `packet_version` | string | Yes | Source packet version. |
| `created_at` | string | Yes | Source response timestamp. |
| `reviewer_name` | string | Yes | Source reviewer display name. |
| `reviewer_kind` | string | Yes | Source reviewer kind. |
| `outcome` | string | Yes | Source review outcome. |
| `source` | string | Yes | Human-readable source label, `review-response packet`. |

`resume_from` identifies the response being consumed. It intentionally does not
copy raw reviewer draft paths or transport metadata.

### `target`

| Key | Type | Required | Notes |
| --- | --- | --- | --- |
| `repository` | string | Yes | Repository owner/name or local project name from `response_to`. |
| `working_branch` | string | Yes | Branch under review from `response_to`. |
| `base_commit` | string | Yes | Fixed lower bound for the reviewed diff. |
| `head_commit` | string | Yes | Fixed upper bound for the reviewed diff. |
| `diff_range` | string | Yes | Reproducible range, such as `base..head`. |
| `pull_request_url` | string | No | Safe PR URL from `response_to` when present. |
| `storage_id` | string | No | Source storage id from `response_to` when present. |

`target` is derived from `review-response.response_to`. The producer does not
expand local paths, GitHub remotes, or raw request prose.

### `tasks[]`

| Key | Type | Required | Notes |
| --- | --- | --- | --- |
| `source_finding_id` | string | Yes | Source review finding id, such as `F1`. |
| `severity` | string | Yes | `high`, `medium`, `low`, or `info`. |
| `blocking` | boolean | Yes | Copied from the finding. |
| `title` | string | Yes | Copied finding title. |
| `description` | string | Yes | Copied finding description. |
| `evidence` | string | Yes | Copied source-linked evidence or reasoning. |
| `recommendation` | string | Yes | Copied concrete requested change or owner decision. |
| `location` | object | No | Copied optional file/line/symbol pointer. |

Tasks are a lossless continuation projection of findings. They are not a claim
that the finding is valid. The receiving implementer still evaluates each task.

### `reviewed_scope`

`reviewed_scope` uses the same shape as `review-response.reviewed_scope`.

### `prior_verification[]`

`prior_verification` uses the same shape as `review-response.verification`.

The different field name is deliberate. It avoids implying those checks were
run after fixes.

### `safety_gates`

| Key | Type | Required | Notes |
| --- | --- | --- | --- |
| `preserve_unrelated_changes` | boolean | Yes | Always `true` for generated packets. |
| `requires_human_approval_for_merge` | boolean | Yes | Always `true` for generated packets. |
| `requires_human_approval_for_publish` | boolean | Yes | Always `true` for generated packets. |
| `requires_human_approval_for_destructive_commands` | boolean | Yes | Always `true` for generated packets. |

## Derived `resume_status`

| Source `review-response.outcome` | `resume_status` |
| --- | --- |
| `approved` | `owner_decision` |
| `changes_requested` | `address_findings` |
| `commentary` | `continue_with_context` |
| `blocked` | `blocked` |

The naming is intentionally not `ready_to_merge`. An approval can still require
owner, CI, branch protection, release, or other project gates.

## Semantic Rules

Schema validation is not enough. The registry semantic check must enforce:

- `resume_status: "address_findings"` requires at least one task with
  `blocking: true`.
- `resume_status: "owner_decision"` forbids blocking tasks.
- `resume_status: "continue_with_context"` forbids blocking tasks.
- `resume_status: "blocked"` requires at least one reviewed-scope limitation.
- `tasks.length` must equal the source response `findings.length` when the
  packet is built by the producer.

The last rule is producer-tested rather than schema-testable because the source
response is not embedded in the resume packet.

## Markdown Rendering

A human-readable render uses this order:

1. Resume project
2. Resume from
3. Target
4. Status and confidence
5. Summary
6. Tasks
7. Reviewed scope
8. Prior verification
9. Safety gates
10. Provenance
11. Redactions
12. Sensitive data
13. Next action

Tasks render as readable blocks, not table rows, for the same reason
`review-response.findings` render as blocks: descriptions, evidence, and
recommendations are prose fields.

Markdown rendering must use the existing shared helpers for table escaping,
code-span text, inline normalization, and label rendering.

## CLI Behavior

```text
open-relay generate resume-project --response <review-response.json> [--format json|markdown] [--output <path>]
```

Behavior:

1. Parse arguments strictly.
2. Reject missing, unknown, duplicate, or invalid flags with exit code `2`.
3. Read the response JSON.
4. Validate it with generic packet validation.
5. Require `packet_type: "review-response"` and `packet_version: "0.1"`.
6. Build a `resume-project/0.1` packet.
7. Validate the generated packet.
8. Print JSON by default or Markdown with `--format markdown`.
9. Write to `--output` when supplied.

Error posture:

- invalid JSON: `Invalid JSON in <path>`; existing JSON parser behavior is
  acceptable here because the path is user-supplied and prior commands already
  use that message shape;
- non-response packet: `Resume-project generation requires a review-response packet.`;
- generated schema failure: `Generated resume-project packet failed validation.`;
- write failure: `Could not write resume-project packet.`;
- output success: `Wrote resume-project packet.`;
- Markdown output success: `Wrote resume-project Markdown.`;
- errors must not echo response contents.

## Relationship To Existing Commands

- `transport github-pr fetch` remains the way to fetch exact packets from a PR.
- `generate resume-project` consumes a local response packet file.
- `render <resume-project.json> --template codex` remains the way to produce a
  Codex-oriented prompt.
- No command in this slice posts, updates, fetches, or deletes GitHub comments.

## Examples

Implementation should add:

- `examples/resume-project/relay.json`
- `examples/resume-project/relay.md`

The example should derive from `examples/review-response/relay.json` and should
be snapshot-bound so renderer drift is caught.

## Security And Data Discipline

- The producer copies only fields from the validated `review-response` packet.
- It does not read git, GitHub, environment variables, package registries, or
  local project files beyond the explicit response path.
- It does not expand local paths or remote URLs.
- It does not run tests or commands.
- It does not invoke Codex, Claude, or any other agent.
- It does not merge, publish, rebase, reset, delete, or write repository source
  files.
- Packet-authored prose remains untrusted context when rendered into a prompt.
- The Codex prompt template must continue to tell the receiver to evaluate
  findings before applying them and to preserve unrelated user changes.

## Lifecycle Coverage

| Lens | Decision |
| --- | --- |
| Create/import | Created locally from a validated `review-response/0.1` file. |
| List/search/view | Viewed through `validate`, generic `render`, and examples; no storage list command. |
| Edit/update | Not edited by Open Relay; regenerate from the source response. |
| Activate/archive/delete | Deferred; no storage or retention behavior in this slice. |
| Ownership | Local CLI user owns the generated file; reviewer owns source review content. |
| Permissions/scope | Local file read/write only; no GitHub or registry access. |
| Audit/events | Git history, generated packet, and command output are the local evidence. |
| Notifications | Deferred. |
| Billing/quota | N/A. |
| Error/empty/recovery/smoke | Empty findings and empty verification are valid; invalid source response fails closed. |

## Non-Goals

- Automatic fixes.
- Automatic test execution.
- Automatic merge, publish, rebase, reset, or destructive commands.
- Agent invocation.
- GitHub fetch/post/update composition.
- Native GitHub review import.
- Response or resume packet storage under `.open-relay/`.
- Implementation-handoff packet type.
- Custom user-authored prompt templates.
- Packet version `0.2`.

## Review Focus Questions

1. Is `resume-project` useful enough as a packet type now that
   `render --template codex` already exists?
2. Is deriving tasks from findings faithful, or does it over-transform reviewer
   content?
3. Are the `resume_status` names automation-safe enough, especially
   `owner_decision` instead of a merge-ready label?
4. Should `prior_verification` copy response verification, or should it be
   omitted to avoid confusing review-time checks with post-fix checks?
5. Does the slice correctly avoid external orchestration while still closing
   the packet-native loop?
